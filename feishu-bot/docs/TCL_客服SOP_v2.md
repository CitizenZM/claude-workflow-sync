### 2.3 数据录入规范

**客诉新表字段填写标准**:
- **shopify单号**: 纯数字，不加 # 号 (例: `1207` 而非 `#1207`)
- **问题类型**: 使用下拉菜单 → `产品缺陷` / `物流破损` / `退货` / `物流异常` / `地址报错` / `其他`
- **解决方案**: `退款` / `换货` / `无退货退款` / `Price Match`
- **当前状态**: 使用下拉菜单（数字编码）→ 见 §3.2 状态码
- **跟进记录**: 每次操作新起一行，格式 `M/D 操作内容`，例：`3.10 催label` `3/15 已退款，待告知顾客`
- **填写日期**: 首次录入日期 (非下单日期)
- **最后更新时间**: 系统自动或手动更新 `MM/DD/YY HH:MM`

**退货 Tab 字段标准**:
- **是否已系统登记(Y/N)**: 退货是否已在领星/Shopify 登记
- **Label是否24h内(Y/N)**: 从客户申请到 label 生成是否在 24h 内 (SLA 指标)
- **至24h内处理(Y/N)**: 从仓库确认收货到退款处理是否在 24h 内 (SLA 指标)

## 3. 工单生命周期管理

### 3.1 工单流转全流程

```
客户邮件/Shopify 后台发现问题
    ↓
客服录入 [客诉新表] → 填写基本信息 + 问题类型
    ↓
判断问题类型 → 路由到对应流程 (§4-§11)
    ↓
执行处理 → 每步更新 [跟进记录]
    ↓
需要退货？→ 同步录入 [退货] Tab
需要破损理赔？→ 同步录入 [不要改数据_破损记录] Tab
需要退款/补发？→ 录入 [退款重发待处理] Tab
    ↓
LTL 订单？→ 同步更新 [LTL Shipping Sheet]
    ↓
最终解决 → 更新状态为 "已解决"
    ↓
关闭工单 → 确认所有关联 Tab 状态一致
```

### 3.2 工单状态码 (当前状态字段)

| 状态码 | 含义 | 适用场景 |
|--------|------|----------|
| 空白 | 新建/处理中 | 初始录入 |
| 2要求包装照片 | 等待客户提供包装破损照片 | 物流破损 |
| 4等待return label | 已申请 label，等仓库生成 | 退货 |
| 5发出return label | Label 已发给客户 | 退货 |
| 5.1上门取货时间 | 等待 LTL pickup 安排 | 大件退货 |
| 6客户寄回 | 客户已寄出，等仓库确认 | 退货 |
| 7待补发 | 待仓库发出替换产品 | 换货 |
| 8已邮寄 | 补发/替换品已发出 | 换货 |
| 9Refund | 已完成退款 | 退款 |
| 10投诉要钱 | 客户发起 chargeback/dispute | 争议 |

### 3.3 关键时间节点

| 节点 | SLA 目标 | 实际表现 (基于历史数据) |
|------|----------|-------------------------|
| 首次回复 | 24h 内 | 大部分达标 |
| Return label 生成 | 48h 内 (目标) | **3-10+ 天** (痛点) |
| 仓库确认收货 | 收货后 24h | 1-7 天不等 |
| 退款执行 | 仓库确认后 24h | 基本达标 |
| 客户通知 | 退款后当日 | **经常遗漏**，跟进记录常见"待告知顾客" |

## 4. 退货退款处理流程

### 4.1 标准退货 (客户主动退货)

**适用**: 客户收货 30 天内不满意/不需要

**费用结构**:
| 产品类型 | Restocking Fee | 退货运费 | 说明 |
|----------|---------------|---------|------|
| TV 电视 (全系列) | **15%** | 客户承担 | 含 QM6K/QM7K/QM8K/X11L/A300W 等 |
| 手机 / 平板 | **5%** | 客户承担 | 含 NXTPAPER/T705M 等 |
| 重复下单 | **0%** (酌情免除) | 视情况 | 需确认非客户操作失误 |

**详细流程**:
```
1. 确认订单在 30 天内 (以 Shopify "Paid at" 为准)
2. 告知退货政策：restocking fee + 客户承担退货运费
3. 等待客户确认接受
4. 客户确认 → 录入 [退货] Tab → 联系小金申请 return label
5. Label 生成后发送给客户 (附退货步骤说明)
6. 等待客户寄出 → 更新状态为 "6客户寄回"
7. 仓库确认收货 → 核实退回产品完整性
8. Shopify 执行退款 (扣除 restocking fee + 运费)
9. 邮件通知客户退款已处理
10. 更新 [客诉新表] 状态为 "已解决" + [退货] Tab 完成
```

**客户长期未回复处理**:
- 发送退货政策后 7 天无回复 → 发 follow-up 邮件
- 14 天无回复 → 标记 "已解决" (视为客户放弃)
- 历史案例: 订单 #1674 客户 4/28 长时间未回复 → 关闭

**退回物品不完整处理**:
- 历史案例: 订单 #1258 退回只有平板，无配件 → 告知客户需自费寄送配件
- 历史案例: 订单 #1795 退回空包裹 → 拒绝退款，标记异常

### 4.2 发货前取消

**适用**: 订单已下但未发货

```
1. 确认 Shopify Fulfillment Status = "unfulfilled"
2. 领星确认未出库
3. 取消订单 → Shopify 全额退款
4. 邮件通知客户
5. 如已出库但未揽收 → 尝试物流拦截
6. 拦截成功 → 全额退款
7. 拦截失败 → 按标准退货流程处理 (含 restocking fee)
```

**历史参考**: 订单 #1863、#1925、#1883 — 发货前取消，直接退款

### 4.3 发货后取消

**费用**: Restocking fee + 往返运费由客户承担
**流程**: 同标准退货，但需明确告知客户额外费用

**历史参考**: 订单 #1695 — 客户坚持取消已发出订单，最终 4.22 退款

### 4.4 缺货退款

**适用**: ZNT 仓库缺货，无法按时发货

```
1. 确认缺货状态 (领星查库存)
2. 给客户两个选项:
   - 方案 A: 等待 15 个工作日 + 赠送 5% 折扣券
   - 方案 B: 全额退款
3. 客户选择后执行
```

**历史参考**: 订单 #2021 — 98QM7K 缺货，客户选择退款

## 5. 物流破损与理赔流程

### 5.1 物流破损处理 (Shipping Damage)

**发现破损时的完整流程**:

```
1. 客户报告破损
    ↓
2. 要求客户提供:
   - 产品破损照片
   - 外包装破损照片 (必须！仓库理赔需要)
   - 如产品无法开机 → 要求提供视频
    ↓
3. 录入 [不要改数据_破损记录] Tab:
   - 破损类型: "包装" 或 "产品"
   - 处理方式: "退款" 或 "补发"
    ↓
4. 判断是否需要客户退回:
   - 包装破损但产品可用 → 可能无需退回 (仓库跟物流理赔)
   - 产品严重损坏 → 需要退回
   - 仓库判断无需退回 → 直接退款/补发
    ↓
5. 需要退回 → 生成 prepaid return label → 安排 pickup (大件) 或客户 drop off (小件)
    ↓
6. 仓库收货确认 → 执行退款或补发
    ↓
7. 录入所有 Tab + 通知客户
```

### 5.2 破损类型与处理策略

| 破损类型 | 处理方式 | 是否需要退回 | 客户费用 |
|----------|----------|-------------|---------|
| 外包装破损，产品完好 | 无退货退款 (仓库理赔) | 否，但客户需保留产品 | 0 |
| 外包装+产品均破损 | 退款或换货 | 是 | 0 |
| 产品缺陷 (非运输) | 退款或换货 | 是 | 0 |
| 无外包装可证明 | 客户自行承担 | — | 全额 |

**关键规则**:
- **必须获取外包装照片**才能走物流理赔 → 历史案例 #1575: 已无外包装，客户自行承担
- 仓库理赔流程中**客户需保留产品**不丢弃 → 历史案例 #1391
- 如客户已发起 chargeback/dispute → 标记状态 "10投诉要钱"，转 Mingyi

### 5.3 历史高频破损产品

基于 破损记录 Tab 数据:

| 产品 | 破损频率 | 常见问题 |
|------|---------|---------|
| 75QM6K / 85QM7K | 高 | 运输内屏破裂、边角碰撞 |
| Q85H | 中 | 边角摔坏、杂音 |
| 65QM8K | 中 | 屏幕弯曲 |
| T705M | 低 | 不是运输问题，多为产品不符预期 |

## 6. LTL 大件物流管理

### 6.1 LTL 承运商

| 承运商 | 覆盖产品 | 特点 |
|--------|---------|------|
| **Daylight** | 85"/75"/98" 电视 | 主力 LTL carrier，BOL tracking |
| **Fedex** (LTL) | 85"/75"/98" 电视 | 302xxx 开头 tracking |
| **ODFL** | 75"/98" 电视 | 78xxx 开头 tracking |
| **GLS** | 85" 电视 | 564xxx 开头 |

### 6.2 LTL Tracking 管理

**问题**: LTL 订单经常无自动 tracking number，需手动处理

**流程**:
```
1. 仓库发货 → 生成 BOL/tracking
2. 小金录入 [TCL LTL Shipping Sheet] 对应月份 Tab
3. 同步更新 Shopify 订单 tracking
4. 同步更新领星发货状态
5. 如客户询问物流 → 先查 LTL Sheet → 提供 tracking
```

**LTL Sheet 字段**:
- 订单号 | 下单日期 | 订单产品 | 发货信息 | 物流渠道 | 追踪码 | 面单生成日期 | 物流网址

**当前数据**: May/2026 有 15 条 LTL 记录，Apr/2026 有 40+ 条

### 6.3 LTL 退货流程

```
1. 确认退货原因 → 生成 return BOL/label
2. 安排 LTL pickup:
   - 通知客户等待物流公司联系预约上门时间
   - 客户需在家配合取货
   - 如首次 pickup 失败 → 重新安排时间
3. 物流取货 → tracking 更新
4. 仓库收货确认
5. 执行退款/换货
```

**历史痛点**:
- 订单 #1832: 4.22 重新发 label，4.28 没有 pickup，重新安排时间，5.4 才到仓库
- LTL pickup 安排周期: **1-2 周**，远超小件快递

### 6.4 LTL 签字配送

- 大件 TV 需要签字确认配送
- 客户可要求更改配送地址或更改签字配送要求
- 历史案例 #1779: 改签字配送，直接处理

## 7. 地址异常处理流程

### 7.1 地址报错分类

基于历史数据，地址报错是**第二高频问题类型** (仅次于退货):

| 场景 | 处理方式 | 举例 |
|------|----------|------|
| **领星地址报错** | 联系客户验证正确地址 → 更新领星 → 发货 | #1376, #1742, #1711, #1735 |
| **客户要求更改地址** | 确认新地址 → 同步仓库 → 如未发货直接改，已发货联系 carrier | #1470, #1592 |
| **PO Box 地址** | Shopify Flow 可能拦截 → 确认客户是否有替代地址 | #1711 (No PO BOX) |
| **公寓/高层补充信息** | 添加 unit/floor/building 信息 → 提交 carrier 特殊配送请求 | #1515 (3rd floor condo) |
| **Business name 缺失** | 补充公司名称 → 更新发货信息 | #1767 (Paul Hastings LLP) |

### 7.2 地址验证标准流程

```
1. 领星/Shopify 标记地址异常
2. 邮件联系客户验证地址:
   "We noticed a potential issue with your shipping address.
    Could you please confirm the correct address?"
3. 客户回复后 → 更新领星 + Shopify
4. 确认地址完整: Street + Apt/Unit + City + State + ZIP
5. 通知仓库按更新地址发货
6. 更新 [客诉新表] 跟进记录
```

### 7.3 地址格式规范

记录地址时必须包含:
```
[Recipient Name]
[Street Address] [Apt/Suite/Unit]
[City], [State] [ZIP]
United States
```

## 8. 价格保护 (Price Match)

### 8.1 政策规则

| 条件 | 规则 |
|------|------|
| 时限 | 购买后 **30 天**内 |
| 适用渠道 | **仅限 DTC 订单** (naec.store 直接下单) |
| 不适用 | Amazon、BestBuy、Walmart 等其他渠道购买 |
| 计算方式 | 差价 = 原价 - 当前价，包含税差 |
| 执行方式 | Shopify partial refund |
| 审批 | 客服可直接执行，无需 Mingyi 审批 |

### 8.2 操作流程

```
1. 客户发邮件要求 price match
2. 确认:
   - 订单来自 naec.store
   - 在 30 天内
   - 当前官网价格确实更低
3. 计算差价 (含税)
4. Shopify → Orders → 找到订单 → Refund → Partial refund → 填入差额
5. 邮件通知客户
6. 录入 [客诉新表]，解决方案 = "Price Match"
```

- 线下渠道购买的产品
- 蓝牙/WiFi 连接问题

禁止:

## 10. 欺诈检测与重复订单

### 10.1 Shopify Flow 自动检测

| 触发条件 | 动作 |
|----------|------|
| PO Box 地址 | 标记 hold → 人工审核 |
| 同客户 24h 内多笔订单 | 标记重复 → 人工审核 |

### 10.2 重复下单处理

```
1. 确认是否真正重复 (客户可能有意购买多台)
2. 确认重复 → 取消多余订单 → 全额退款
3. 免除 restocking fee (网站原因导致)
4. 如客户声称网站问题 → 免除所有费用
```

**历史参考**: #1080 重复下单退款；#1182/#1183 客户认为双重下单是网站原因，免除 restocking fee

### 10.3 Chargeback/Dispute 处理

- 客户发起 dispute → 标记状态 "10投诉要钱"
- **不要主动退款** → 等银行裁决
- 提供 Shopify 订单记录 + 物流签收证明给银行
- Mingyi 决策是否接受 dispute
- 历史案例: #1391 客户与银行 dispute；#1808 Fedex 送达但客户否认收货

## 附录 C: 退货 Tab 状态流转

```
退货申请 → 系统登记 (Y) → 申请 Label → Label 生成 → 发给客户
    → 客户寄回 → 物流签收 → 仓库确认 → 开箱检查 → 处理退款/补发
    
状态值:
- 等label中: 已申请，等仓库生成
- 顾客寄回: 客户已寄出
- 等待return label: label 还没申请
- 等上门取货时间: LTL pickup 安排中
- 已邮寄: 补发品已发出
- Refund: 退款完成
```
